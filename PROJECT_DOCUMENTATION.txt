PROJECT OVERVIEW
================================================================================

HealthSaathi is an AI-powered health companion platform designed to provide 
instant medical triage, nearby health facility discovery, and personalized 
medical guidance. The system is built with a multilingual interface (Nepali 
and English) and serves users across 77 districts in Nepal.

Core Features:
  • AI-powered symptom triage and risk assessment
  • Medical image generation from text prompts using DALL-E
  • Vision-based image analysis for medical report interpretation
  • PDF extraction and medical document analysis
  • Facility location discovery using GPS mapping
  • Multilingual support (Nepali and English)
  • Conversation history management
  • Text-to-Speech health guidance
  • User authentication and profile management
  • Real-time medical consultation support


TECHNOLOGY STACK
================================================================================

BACKEND:
  • Framework: Django 4.2.29 (Python web framework)
  • REST API: Django REST Framework 3.16.1
  • Database: PostgreSQL (via psycopg2-binary)
  • Vector Database: ChromaDB 1.5.5 (for AI embeddings and similarity search)
  • Mapping API: Baato 1.0.1 (location and facility mapping)
  • Image Processing: Pillow 10.4.0 (image manipulation), pdf2image (PDF extraction)
  • Document Processing: PyPDF2 3.0.1 (PDF text extraction)
  • Vision API: Azure OpenAI Vision for image analysis
  • Image Generation: Azure OpenAI DALL-E for medical image generation
  • Server: Gunicorn 20.1.0 (production WSGI server)
  • CORS: django-cors-headers 4.9.0 (cross-origin requests)
  • Environment: python-dotenv 1.2.2 (environment configuration)
  • HTTP Client: requests 2.32.5 (external API calls)

FRONTEND:
  • UI Framework: React 18.2.0 (modern component-based UI)
  • Build Tool: Vite 7.3.1 (fast module bundler)
  • Styling: Tailwind CSS 3.3.5 (utility-first CSS framework)
  • Routing: React Router DOM 6.20.0 (client-side navigation)
  • HTTP Client: Axios 1.6.0 (API communication)
  • Mapping: Leaflet 1.9.4 + React Leaflet 4.2.1 (interactive maps)
  • CSS Processing: PostCSS 8.4.31 + Autoprefixer 10.4.16

INFRASTRUCTURE:
  • Version Control: Git
  • Development: Node.js ecosystem for frontend

CODE FLOW AND ARCHITECTURE
================================================================================

USER AUTHENTICATION FLOW:
1. User accesses LoginPage.jsx
2. Enters email and password credentials
3. Frontend calls triageService.login() → api.post('/login/')
4. Backend LoginView validates credentials against User model
5. Django generates authentication token
6. Frontend stores token in localStorage
7. User redirected to ChatPage.jsx
8. AppContext maintains authenticated user state globally

SYMPTOM TRIAGE FLOW:
1. User enters symptoms in ChatPage.jsx
2. Frontend calls triageService.sendTriage(symptoms, location)
3. API request sent to backend TriageView with:
   - User symptoms (text)
   - User location (latitude, longitude)
   - User context from profile
4. Backend TriageView processes:
   - Validates user authentication
   - Creates TriageSession record in database
   - Sends symptoms to AI model (ai_client.py)
   - AI analyzes and generates:
     * Risk level (HIGH/MEDIUM/LOW)
     * Brief medical advice
     * Detailed health guidance
     * Dietary recommendations (food_eat, food_avoid)
     * Dos and Don'ts
   - Uses ChromaDB for semantic search of similar symptoms
   - Returns structured JSON response
5. Frontend receives response and displays RiskCard
6. User sees:
   - Risk assessment with color coding
   - Medical advice in current language
   - Facility recommendations
   - Text-to-speech playback option

IMAGE GENERATION FLOW:
1. User enters image description prompt in ChatWindow.jsx
2. Clicks image generation button to open modal
3. Frontend calls imageService.generateImage(prompt, size, quality)
4. Client-side rate limiter checks: 3 requests per minute max (20 sec interval)
5. Backend VisionAnalysisView processes:
   - Validates prompt content
   - Calls Azure OpenAI DALL-E API
   - Falls back to HuggingFace Stable Diffusion if needed
   - Saves generated image to ./image_storage directory
   - Returns base64-encoded image data
6. Frontend displays generated image in chat
7. User can:
   - View full-size image in message
   - Download generated image
   - Include medical images with consultations

IMAGE ANALYSIS FLOW:
1. User uploads medical image via image upload button
2. Frontend converts image to base64 via fileToBase64()
3. User asks analysis question in chat
4. Frontend calls imageService.analyzeImage(imageBase64, question)
5. Backend VisionAnalysisView processes:
   - Sends image + question to Azure OpenAI Vision API
   - Generates medical analysis of the image
   - Returns structured analysis text
6. Frontend displays analysis in MessageBubble
7. User sees:
   - Original uploaded image
   - AI-generated medical analysis

PDF PROCESSING FLOW:
1. User uploads PDF medical report via (previously available)
2. Frontend calls imageService.processPDF(pdfFile)
3. Backend PdfProcessingView processes:
   - Extracts images from PDF pages
   - Extracts text content from PDF
   - Analyzes each page image with Vision API
   - Generates consolidated summary of findings
4. Frontend displays:
   - File metadata (pages, extracted images count)
   - Page-by-page medical analysis
   - Overall summary and conclusions
5. User can:
   - Review extracted text
   - See Vision API analysis per page
   - Get automatic medical report interpretation

LOCATION-BASED FACILITY DISCOVERY:
1. ChatPage requests user location via locationService.getCurrentLocation()
2. Browser Geolocation API returns latitude/longitude
3. locationService.getPlaceAndCity() calls backend location endpoint
4. Backend uses Baato API to:
   - Reverse geocode coordinates → place name and district
   - Fetch nearby health facilities (hospitals, clinics)
5. Frontend displays:
   - Current location with district name
   - Interactive map with facility markers
   - Facility details (name, distance, type)
6. Facilities specific to user's district shown in triage advice

INSTALLATION AND DEPLOYMENT
================================================================================

BACKEND SETUP:
  1. Navigate to backend directory
  2. Create virtual environment: python -m venv venv
  3. Activate: source venv/bin/activate (Unix) or venv\Scripts\activate (Windows)
  4. Install dependencies: pip install -r requirements.txt
  5. Configure environment variables in .env file:
     - Set AZURE_OPENAI_API_KEY, AZURE_IMAGE_ENDPOINT, AZURE_VISION_ENDPOINT
     - Set IMAGE_STORAGE_DIR=./image_storage (Windows-compatible path)
     - Set IMAGE_RPM_LIMIT=3 and IMAGE_MIN_INTERVAL_SECONDS=20
  6. Create image and log directories: mkdir image_storage
  7. Run migrations: python manage.py migrate
  8. Test image endpoints: curl -X POST http://localhost:8000/api/image/generate/
  9. Start server: python manage.py runserver
  10. Production: gunicorn nepalcare.wsgi (with image storage on persistent volume)

Image Feature Configuration:
  • Verify Azure OpenAI account has image generation permissions
  • Test AZUIRE_IMAGE_ENDPOINT and AZURE_VISION_ENDPOINT connectivity
  • Ensure image_storage directory has write permissions
  • Monitor disk space usage (images stored locally)
  • Set up log rotation for generated image logs

FRONTEND SETUP:
  1. Navigate to frontend directory
  2. Install dependencies: npm install
  3. Configure API base URL in src/services/api.js
  4. Set rate limit constants in imageService.js (defaults: 3/min, 20 sec apart)
  5. Configure image generation modal styling if needed
  6. Development: npm run dev
  7. Build: npm run build
  8. Test image features in browser dev tools
  9. Production: Deploy build/ folder to static hosting (with API proxy for /api/image/)

Testing Image Features:
  • Test image generation with valid prompts
  • Upload sample medical images for vision analysis
  • Verify rate limiting with rapid requests
  • Check browser console for imageService logs
  • Verify image_storage directory receives generated files

  • API key rotation for Azure OpenAI endpoints


IMAGE PROCESSING CONFIGURATION
================================================================================

Rate Limiting:
  • Global Limit: 3 image requests per minute per user
  • Minimum Interval: 20 seconds between image generation requests
  • Vision API: 10 requests per minute (6 seconds between requests)
  • Enforcement: Client-side (JavaScript) + Server-side (Python)
  • Purpose: Prevent API quota exhaustion and excessive costs

Image Storage:
  • Directory: ./image_storage/ (relative path, cross-platform compatible)
  • Contents: Generated images, extracted PDF images, processing logs
  • Cleanup: Manual purge recommended every 30 days
  • Disk Usage: Monitor to prevent capacity issues

API Endpoints (Azure OpenAI):
  • Image Generation: AZURE_IMAGE_ENDPOINT (DALL-E 3)
  • Vision Analysis: AZURE_VISION_ENDPOINT (GPT-4 Vision)
  • Authentication: AZURE_OPENAI_API_KEY
  • Fallbacks: HuggingFace Stable Diffusion, Pollinations API

Environment Variables Required:
  • AZURE_OPENAI_API_KEY: Primary API authentication
  • AZURE_IMAGE_ENDPOINT: Image generation endpoint URL
  • AZURE_VISION_ENDPOINT: Vision analysis endpoint URL
  • IMAGE_STORAGE_DIR: Path for file storage (./image_storage for Windows)
  • IMAGE_RPM_LIMIT: Rate limit per minute (default: 3)
  • IMAGE_MIN_INTERVAL_SECONDS: Minimum seconds between requests (default: 20)

CLIENT-SIDE IMAGE SERVICE (src/services/imageService.js):
  
  Main Functions:
    
    generateImage(prompt, size, quality, n)
      - Generates medical image from text
      - Client rate limiting enforced
      - Returns Promise<{images, source, error}>
    
    analyzeImage(imageSource, question)
      - Analyzes medical image with Vision AI
      - imageSource: base64 or URL
      - Returns Promise<{analysis, source, error}>
    
    processPDF(pdfInput)
      - Handles PDF file or path
      - Returns Promise<{total_pages, extracted_images, text_content}>
    
    getLocalRateLimitStatus()
      - Returns {limit, rateLimited, waitSeconds}
      - Used for UI feedback without server call
    
    fileToBase64(file)
      - Converts File object to base64 string
      - Supports all image formats
      - Returns Promise<string> (data URI)
    
    downloadImage(imageUrl, filename)
      - Downloads generated image to client
      - filename: "generated_image.png"
  
  Rate Limiter Implementation:
    - Tracks timestamps of all requests (last 60 seconds)
    - Enforces minimum 20 second interval between requests
    - Max 3 requests in any 60-second window
    - Returns estimated wait time if throttled
    - Client applies automatic backoff before server error

  Rate Limit Errors:
    • "⏱️ Rate limit in effect. Please wait X.Xs before generating..."
    • Client blocks UI until rate limit passes
    • Server also enforces to prevent bypass
    • Clear messaging about 3 requests per minute limit
  
  API Errors:
    • 404: Endpoint not found - suggests Django restart
    • 401: Authentication failed - check API key configuration
    • Rate limit exceeded: Auto-retry after wait period
    • Image format not supported: List supported formats
    • PDF corrupted: Offer file validation suggestions
  
  Vision API Failures:
    • Fallback to alternate analysis approach
    • Return best-effort analysis with error flag
    • User sees "Analysis confidence: LOW" if partial failure
  
  Image Generation Failures:
    • Primary: Azure OpenAI DALL-E
    • Fallback: HuggingFace Stable Diffusion (free)
    • Last Resort: Pollinations.ai free API
    • User sees which provider generated the image
